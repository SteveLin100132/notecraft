# **AI 導入與專案／產品管理討論紀錄**

**時間：** 2026-06-11 10:05–12:05  
**主題：** AI 輔助開發流程、規格撰寫、版本控管、測試驗證與新 HR 系統技術架構討論

## **一、會議摘要**

本次討論聚焦在如何把 AI 導入既有工程與產品開發流程中，包含 PRD／規格文件撰寫方式、AI 產出 Prototype 的可靠性、Git Flow 與版本控管、AI Agent／Skill／Sub-agent 的使用邊界，以及新 HR 系統的技術架構選型。

整體共識是：AI 可以大幅加速規格整理、Prototype 產出、測試腳本撰寫與初步開發，但不能把需求判斷、驗收與架構決策完全交給 AI。AI 容易根據常見模式「猜」出看似合理的結果，也容易過度設計，因此規格、驗收標準、Git 紀錄與人工驗證會變得更重要。

## **二、重點整理**

### **1. PRD 與規格文件的寫法**

目前的做法是先由人手動撰寫 PRD，定義角色、使用者、功能目標、業務流程、限制範圍、卡控機制與驗收標準，再交給 AI 協助產出畫面或 Prototype。

排班功能的 PRD 內容包含：

- 管理員、負責人、資深人員等角色定義
- 資深人員填寫不能排班的時間
- 負責人依據排除時間進行排班
- 國定假日、休假日、補班日等特殊情境
- 活動設定、工作設定、草稿、一鍵產生排班
- 初版自動推薦排班邏輯，讓人員分配盡量平均

初步測試結果顯示，完整 PRD 丟給 AI 後，可以產出約九成符合預期的設計；若只用一句較模糊的需求，AI 也能產出約八成可用的功能。但這不代表詳細規格沒有價值，因為這類排班系統屬於常見模式，AI 可以靠通用知識猜出大致結構。若換成薪資、多國法規、台灣在地勞動規則等複雜情境，模糊需求會更容易偏離預期。

### **2. AI 產出的風險：猜測、偏歐美資料、過度設計**

討論中提到，AI 模型的訓練資料仍偏向英文與歐美脈絡。若處理台灣在地法規、人資流程、薪資計算、勞動規範等需求，不能假設 AI 會自然符合台灣情境。

主要風險包括：

- AI 會用「常見系統通常怎麼做」去猜需求
- 中文與台灣資料比例相對少，容易偏向歐美邏輯
- 遇到複雜需求時，可能沒有真的讀完整規格
- 有時會自作聰明，自行補出使用者沒有要求的東西
- 規劃型任務容易過度設計，產出太多實際用不到的功能

因此後續規格要更重視「明確性」與「驗收標準」，而不是只追求文件內容多。

### **3. AI 工具與 Session 使用方式**

討論到 Claude、Codex、Fable、OpenAI 等工具的使用差異。大方向是，不應完全依賴模型能力，因為模型表現可能受到算力、使用量、供應商策略影響，同一工具在不同時段的品質也可能不同。

使用 AI Session 時的建議：

- 單一 Session 不要塞太大的任務
- 不要在重要開發 Session 中穿插太多無關問題
- 延伸問題可以先記到 Task，再另開或後續處理
- 大任務應拆成小任務，必要時用 Sub-agent 平行處理
- Sub-agent 適合資料蒐集、局部任務、獨立檢查，避免污染主 Session context
- Workflow 類功能若一次派太多 Agent，容易快速消耗額度，不建議一開始就重度使用

### **4. Skill、Sub-agent 與 Agent 的角色分工**

討論中對 Skill 與 Sub-agent 的定位有初步區分：

- **Skill** 比較適合定義固定流程、檢查規則、工作方法
- **Sub-agent** 比較像預先設定好的角色，可以處理特定類型任務
- **Agent loop** 適合 MVP 或 Prototype 階段，但不適合直接作為正式產品的核心執行方式

原因是 Agent loop 成本難預估，正式營運後如果每個流程都靠 Agent 自己查資料、推理、重試，API 成本與穩定性都會失控。比較合理的方式是：Prototype 階段可以先用 Agent 快速驗證；正式產品則要逐步拆成可程式化節點，或縮小成單次 LLM call 可以完成的明確任務。

### **5. Hook 與版本控管**

有討論到用 Hook 自動處理版本號、文件更新時間與更新日誌。

初步結論是：

- Local hook 比較適合當提醒或哨兵，不適合直接讓 AI 自動改版本號
- 版本號若要和部署、Docker Tag、Release 對應，最好放到 GitHub Action 或 CI/CD 流程處理
- AI 可以協助判斷這次更新是否需要調整版本，但不應完全自動放行
- 文件版本、更新時間、更新日誌可以先用指令讓 AI 根據 Git diff 產生摘要，再由人確認

版本號討論中也提到，不同專案可以有不同策略。若只是部署識別，版本號可能只是辨識不同 build；若要支援 rollback、Docker tag、正式 release，則版本規則需要更嚴謹。

### **6. 測試與驗證策略**

共識是 AI 可以加速開發，但驗證責任不能消失。

建議流程：

1. 需求與規格先明確化
2. AI 協助產生開發文件或 Issue
3. AI 開發功能
4. 單元測試與整合測試先跑
5. 使用另一個 Session 或另一個 Agent 做交叉檢查
6. 最後仍需人工做 End-to-End 驗證

特別提醒：AI 有可能產生「看起來有測，但其實沒有涵蓋核心情境」的測試。測試通過不等於需求正確，因此驗收情境要從規格階段就先定義清楚。

網頁或介面流程可以考慮用 Playwright 撰寫自動化測試腳本，後續整合到 CI/CD。AI 可以協助產生腳本，但最後結果仍需人工確認。

### **7. Issue、Git Flow 與文件關聯**

討論中認為，在 AI 開發時代，Git Flow 反而更重要。

原因是 AI 不一定會主動翻過往 Spec，但比較可能讀 Codebase、Commit、PR 與 Issue。若需求、修改原因、技術決策都留在 Issue 與 Commit message 裡，未來 AI 要追查某段程式為什麼被改過，會更容易找到脈絡。

建議做法：

- 新功能應開 Issue，並列清楚背景、需求、技術規格、驗收標準
- 小修正或急件可以簡化流程，但仍盡量保留 PR 或 Commit 紀錄
- Commit message 可以讓 AI 協助寫清楚
- Spec 文件偏向人看的依據；Issue、PR、Commit 則是 AI 與工程流程追溯的重要資料
- 文件內可建立詞彙表與超連結，降低名詞歧義

### **8. 專案管理模式：Waterfall、Agile 與混合模式**

討論到專案與產品的差異：

- **專案**：時間、資源、需求相對明確，完成後可能結束
- **產品**：需求會持續變動，需要反覆實驗、驗證與創造商業價值

傳統上，需求明確的專案可以偏 Waterfall；產品型開發則偏 Agile。但在 AI 輔助開發下，界線會變得模糊，因為需求整理、Prototype、系統分析、系統設計、部分開發與測試可能會被 AI 壓縮到同一個流程中。

目前傾向：

- 新 HR 系統與薪資系統仍需要明確規格與驗收
- 但實作節奏可以偏敏捷，每週小版本迭代
- 版本不要太大，否則驗證成本會過高
- 不建議週末上線，避免出問題時無法即時處理
- 大客戶或高優先級需求仍可能打斷原本 Sprint，這是現實管理問題

### **9. 需求分析方法**

需求分析部分提到可以用 5W1H、流程分支圖、心智圖、狀態圖、測試情境表等方式協助收斂需求。

重點不是把文件寫得很長，而是要釐清：

- 為什麼要做這個功能
- 使用者是誰
- 哪些情境會發生
- 正常流程是什麼
- 例外流程是什麼
- 欄位格式、最大值、最小值、必填規則是什麼
- 日期、時區、多國語系等問題要怎麼處理
- 哪些地方不能讓使用者自由輸入
- 驗收標準是什麼

特別提到 free typing 是高風險來源，因為前端卡控不清楚，後面資料庫與業務流程會很容易出問題。

### **10. 新 HR 系統技術架構討論**

目前初步架構方向是先採單體架構，依 Domain 劃分模組，未來再視情況拆微服務。

前端方向：

- React 生態系
- Tailwind CSS
- shadcn/ui 作為 UI 元件管理基礎
- TypeScript
- Zod 做前後端一致驗證
- Vitest 做單元測試
- Playwright 可作為 E2E 測試
- Storybook 目前未必必要，除非元件狀態複雜、動畫多，或未來多專案共用設計系統

前端狀態與 API 層討論：

- 不建議把 API call 與狀態更新混在同一層 action 裡
- API 存取應抽象成一層，避免未來從原生 fetch、TanStack Query 或其他工具遷移時成本過高
- TanStack Query 的優點是處理 loading、success、error、cache 等 Promise 狀態，但團隊若不熟，也會有學習成本
- AI 產 React 時容易亂加不必要的 hook 與效能優化，需要建立規範

後端方向：

- NestJS
- TypeScript
- Zod 或類似工具做驗證
- Pino 作為 logging library
- Sentry 可作為錯誤監控與通知
- JWT、OIDC 作為認證基礎
- 授權可能需要 ABAC，而不只是傳統 RBAC
- Helmet 等安全 header 應從初期就加上

資料庫與基礎設施：

- PostgreSQL 是主要候選
- Postgres 擴充性較強，未來若有知識庫需求，可評估 pgvector
- ORM 是否使用 Prisma 仍需評估；AI 可能受廣告或常見文章影響而偏向推薦 Prisma
- 也可以考慮 Knex 或直接寫原生 SQL
- Redis 可用於快取與 Queue
- LLM 任務或耗時任務應使用 Queue 控制，例如 BullMQ
- 環境至少應有兩套，理想是 dev／staging／production 三套
- 部署目前可依既有 GCP 或 VPS／Docker 流程評估

### **11. 監控、錯誤處理與資安**

錯誤偵測是後續需要重視的部分。Sentry 的價值在於可以即時通知錯誤、聚合重複錯誤、指出錯誤發生位置，對初期產品穩定性有幫助。

監控可考慮：

- Sentry：前後端錯誤追蹤
- GCP Logging／Monitoring：若沿用 GCP
- Prometheus／Grafana：若走開源監控方案
- Pino：後端結構化 log

資安方面，初期就應處理：

- JWT／OIDC
- 權限模型
- 安全 header
- 基本操作 log
- 使用者操作紀錄
- 未來多公司、多國、多角色時的授權模型

授權模型是風險點，因為 HR／薪資系統不只是登入問題，而是資料可見範圍、公司別、部門別、角色別、法規別與操作權限的組合問題。

## **三、初步共識**

AI 導入不是把工程師或 PM 的判斷拿掉，而是把產出速度提高。因此真正瓶頸會從「寫不寫得出來」轉移到「規格是否清楚、測試是否完整、驗收是否可信、Git 紀錄是否可追溯」。

目前較穩的方向是：

- 規格與驗收標準要先寫清楚
- AI 任務要拆小
- Prototype 可以快，但正式流程要有人驗證
- GitHub Issue／PR／Commit 要成為主要追溯來源
- Skill 與 Sub-agent 可以用，但不能假設它們會永遠自動遵守
- Agent loop 適合 MVP，不適合直接作為 Production 核心流程
- 新 HR 系統先用單體架構，依 Domain 模組化，未來再拆微服務
- PostgreSQL、Redis、Queue、Sentry、TypeScript、NestJS、React／Tailwind 是目前較合理的初步組合
