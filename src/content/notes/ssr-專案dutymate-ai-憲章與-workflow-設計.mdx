# Duty Mate SDD Workflow

> 本文件為 Duty Mate 專案 SDD（Spec-Driven Development）流程的權威說明，涵蓋從需求進入到封存的七個階段，以及每個 Task 的微觀交付迴圈。

**圖例：**

- :badge[手動]{variant="warning" icon="bolt"} — 需要使用者主動下提示詞觸發
- :badge[自動]{variant="info"} — 由上一步自動串接，使用者不用動手
- :badge[新]{variant="success" icon="sparkle"} — 新增角色（相對舊流程）

---

## 1. 角色總覽

```
┌ Skills（流程編排／品質閘，由使用者觸發）
│   openspec-propose · clarify · openspec-apply-change · openspec-archive-change
│   /code-review（tech-lead 借用）
│
├ Subagents — 治理層
│   pm           需求釐清、規格撰寫
│   tech-lead    任務拆解／派工規劃／Code Review／打勾
│
├ Subagents — 分析／設計層
│   sa  ★      全域脈絡＋業務流程 Flow Chart
│   sd  ★      技術設計／UML（Class Diagram、Sequence Diagram）
│   ui-ux ★    前端設計藍圖（逐頁 wireframe＋Design Token 映射，依 Prototype，與 sd 平行）
│
├ Subagents — 實作層
│   backend-developer · dba · ui-ux · technical-writer
│
└ Subagents — 品質層
    qa  ★      驗收審查（Scenario + DoD），不做 Code Review
```

:::warning{title="執行原則"}
subagent 不能自行呼叫其他 subagent；所有跨 agent 的銜接皆由**主 agent**（即與使用者對話的 Claude）依據本文件流程驅動。
:::

### 2.1 例外通道：超小型功能可跳過 sd 與 ui-ux

:::info{title="例外通道"}
若 sa 階段判定為**超小型調整**（如純文案／設定值微調），不涉及新類別、新互動序列，也不涉及版面或元件變動，可在使用者確認後**直接委派 openspec-propose**，跳過 [2a]／[2b]。此屬例外，預設仍走 sa →（sd ∥ ui-ux）→ propose。
:::

### 2.2 change 名稱命名規則

change 名稱由 SA 在步驟 1 從 Spec 檔名推導，所有後續階段使用同一名稱：

- 去掉數字前綴與分隔符號：`01-login.md` → `login`
- **必須以字母開頭**（OpenSpec CLI 限制，數字開頭會被拒絕）
- 使用 kebab-case：`02-schedule-management.md` → `schedule-management`
- SA 透過 `openspec new change "<name>"` 建立目錄（含 `.openspec.yaml`），後續 SD、ui-ux 與 openspec-propose **使用既有目錄**，不重新建立

最終 change 目錄結構：

```
openspec/changes/<name>/
├── .openspec.yaml          ← SA 透過 openspec new change 建立
├── system-analysis.md      ← SA 產出
├── system-design.md        ← SD 產出（後端技術藍圖）
├── system-ui-design.md     ← ui-ux 產出（前端設計藍圖，與 SD 平行）
├── proposal.md             ← openspec-propose 產出
├── design.md               ← openspec-propose 產出
├── tasks.md                ← openspec-propose 產出
└── specs/                  ← openspec-propose 產出
    └── <capability>/
        └── spec.md
```

## 5. 跨 Agent 文件流向

> §2 的互動流程圖點擊任一節點後，右側面板底部會列出「本階段產出被誰必讀」；以下為純文字對照版本。

:::tip{collapsible title="展開純文字版文件流向（ASCII fallback）"}

```
sa    ──→ openspec/changes/<name>/system-analysis.md     ┐
                                                       ├─→ 被 sd 必讀
                                                       ├─→ 被 ui-ux 必讀
sd    ──→ openspec/changes/<name>/system-design.md       │
ui-ux ──→ openspec/changes/<name>/system-ui-design.md    ┘  皆被 tech-lead 必讀
                                                            皆被 openspec-propose 必讀
                                                            system-ui-design.md 亦被實作階段 UI task 必讀

openspec-propose ──→ openspec/changes/<name>/{proposal,design,tasks}.md
                                                  ├─→ 被 tech-lead 必讀
                                                  ├─→ 被實作 agent 必讀
                                                  └─→ 被 qa 必讀（spec 為驗收準繩）

tech-lead ──→ tasks.md（補 @assignee/deps/parallel）
                                                  └─→ 被主 agent 在 openspec-apply-change 階段讀取派工
```

:::

:::warning{title="qa 驗收唯一準繩"}
qa 的驗收**唯一準繩**是 OpenSpec spec 的 Scenario。`system-analysis.md`、`system-design.md` 與 `system-ui-design.md` 僅供脈絡參考，不可取代 Scenario 驗收。
:::

## 9. 與既有規範的對應

- **派工機制**：所有派工資訊（@assignee、deps、parallel 群組）全寫進 tasks.md，由主 agent 在 openspec-apply-change 階段讀取後直接派給對應 subagent。
- **測試分層**：單元（Jest）、整合（Supertest）由 backend-developer 以 TDD 完成；E2E（Playwright）與 Scenario 驗收由 qa 執行（見 [CLAUDE.md §1-2](../CLAUDE.md)）。
- **Commit 時機**：每完成一個 Task（過 Code Review + QA）即 commit；同章節或同前綴編號群組通過後一起 commit（見 [CLAUDE.md §3.2](../CLAUDE.md)）。
- **DoD**：完整清單以 [.claude/agents/qa.md](.claude/agents/qa.md) 為準（見 [CLAUDE.md §2](../CLAUDE.md)）。
- **Design Token**：tech-lead Code Review 須檢查 UI 變更是否引用 `public/css/tokens/` 的 CSS 變數，禁止寫死色值與尺寸（見 [CLAUDE.md §1-4](../CLAUDE.md)）。

### 10.1 通用提醒

:::warning{title="通用提醒（易踩雷）"}

- **不要跳階段**：每階段都需使用者明確觸發；下游 agent 只有在使用者確認上一階段摘要後才會啟動。**[2a] sd 與 [2b] ui-ux 為平行分支**，需兩者都完成、使用者確認後才觸發 [3] propose。
- **平行分支要「一次貼」**：[2a] + [2b] 的提示詞已合併為單一段落，請整段一次貼出，讓主 agent 在同一 turn 內同時發起兩個 subagent 呼叫（parallel tool calls）；拆兩則貼會退化成序列，失去平行意義。
- **不要省略「必讀」**：每階段提示詞都明列前置文件路徑，避免下游 agent 只看對話 context、漏掉持久化文件。UI 相關 task 於實作階段必須指向 `system-ui-design.md`。
- **不要替 agent 打勾**：使用者不直接動 `tasks.md`，打勾權責屬 tech-lead（見 §7）。

:::
