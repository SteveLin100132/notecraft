# Mgmt. Tool（管理工具）

專案管理常用的工具，彼此分工如下：

| 工具 | 回答的問題 | 層級 |
| --- | --- | --- |
| **WBS** | 要做哪些事？（範圍） | 規劃 |
| **Gantt** | 什麼時候做、誰先誰後？（時程與依賴） | 排程 |
| **Kanban** | 現在進行到哪了？（流動狀態） | 追蹤 |
| **Task** | 這件事的細節與紀錄是什麼？（執行） | 執行 |

> 流程上，WBS 拆出任務 → 排進 Gantt 看時程 → 變成 Kanban 卡片流動 → 每張卡片背後對應一個 Task 紀錄。

---

## WBS（Work Breakdown Structure）

> 工作分解結構：把專案的最終交付物，**由上而下層層拆解**成可被管理、估時、指派的工作單元；最末端的單元稱為 **Work Package（任務）**。

### 結構特性

- **樹狀結構**：專案總目標 → 主要交付物 → 子交付物 → 任務
- **100% Rule**：子節點總和必須等於父節點的全部範圍，同時也包含總工時
- **以「交付物 / 成果」為導向**，而非以「動作」為導向（避免拆成一堆動詞清單）

### 拆解的粒度

依 **8/80 法則**，一個任務的工時建議落在 **8 ～ 80 小時**之間：

- **太大** → 難以估時與追蹤，風險被掩蓋
- **太細** → 管理成本超過實際產出價值
- **理想** → 能被一個負責人扛起，且驗收標準明確

### 為什麼 PM 需要 WBS

- **確認範圍（Scope）**：未列入 WBS 的東西，預設就不在這次專案內 → 有效對抗 Scope Creep
- **估算的基礎**：沒有拆解就沒有估時與成本估算
- **風險識別**：拆解過程會浮現原本沒想到的依賴與盲點
- **指派責任**：每個任務都應有明確負責人（RACI 中的 **R**）
- **承接後續工具**：Gantt、Kanban、Task 都是以末端任務為基礎再展開

## Kanban（看板）

> 源自豐田生產系統，核心是**視覺化工作流**與**限制在製品數量（WIP, Work In Progress）**，目標是讓任務在流程中順暢流動，並讓瓶頸無所遁形。

### 基本元素

| 元素 | 說明 |
| --- | --- |
| **Board（看板）** | 代表整個工作流的板子 |
| **Column（欄位）** | 對應工作流的每個階段，如 Backlog / To Do / In Progress / Review / Done |
| **Card（卡片）** | 對應一個 Task，在欄位之間由左往右流動 |
| **WIP Limit** | 每個欄位同時可存在的卡片數量上限 |
| **Swimlane（泳道）** | 橫向切分，區分不同類型工作（如 新功能 / Bug / 技術債）或不同產品線 |

### 核心精神

- **視覺化（Visualize）**：所有人一眼看到目前狀態與卡關處
- **限制 WIP**：強迫團隊「先完成、再開始」，避免同時推太多卻沒一件結案
- **管理流動（Manage Flow）**：關注 **Lead Time**（建立 → 完成）與 **Cycle Time**（開始 → 完成）
- **明確的流程政策**：每個欄位的「進入條件」與「完成定義（DoD）」要寫清楚
- **持續改善（Kaizen）**：定期檢視瓶頸、調整欄位與 WIP 上限

### 為什麼 PM 適合用 Kanban

- **進度透明**：不必開會問進度，看板本身就是進度
- **瓶頸可視化**：哪個欄位卡片堆積，就是當前瓶頸（通常是 Review 或驗收）
- **適合需求持續進來的情境**：相較 Scrum 的固定 Sprint，Kanban 沒有迭代邊界，較適合維運型、客服型或需求變動快的專案
- **與 WBS 搭配**：WBS 拆出的任務，可一張張變成卡片直接進入流動

### Kanban vs Scrum

| 面向 | Kanban | Scrum |
| --- | --- | --- |
| 時間框 | 無固定週期 | 固定 Sprint |
| 變更彈性 | 隨時可加入新卡片 | Sprint 中原則上不變動 |
| 角色 | 無強制規範 | 明確角色（PO、SM、Dev Team） |
| 指標 | Lead Time / Cycle Time / Throughput | Velocity |

### 實務小提醒

- **WIP Limit** 一開始設寬一點，再依實際狀況收斂，比一開始就卡死好
- **Done** 一定要有明確驗收標準，否則會變成「我覺得我做完了」
- 卡片**一定要有負責人**，沒人負責的卡片等於沒人在推進
- 定期清理 **Backlog**，長期沒被動到的卡片要嘛刪掉、要嘛降級成想法清單，否則看板會失去信號

### 我的欄位設計

我自己的 Kanban 欄位直接對應下面 Task 的狀態流：

```
Open ──→ On Going ──→ Done ──→ Verified ──→ Closed
(待辦)    (進行中)    (待驗收)   (待結案)     (已結案)
```

> **Pending** 與 **Cancelled** 我不放成欄位，而是用卡片標籤或泳道處理——因為它們不是流程上的正常往前推進，而是「暫時離開流動」或「退出流動」的狀態。

## Task（任務）

一個任務需要包含足夠資訊，才有助於追蹤管理。

### 時間欄位

- 預計完成起訖時間
- 實際完成起訖時間
- 驗收起訖時間
- 上線時間

### 人員欄位

- 需求提案人
- 需求審核人
- 執行人
- 驗收人
- 上線審核人

### 狀態：我的習慣

| 狀態 | 說明 |
| --- | --- |
| **Open** | 任務建立了，但還沒開始 |
| **On Going** | 正在進行中，尚未完成 |
| **Done** | 已完成，等待驗收 |
| **Verified** | 驗收通過，等待結案 |
| **Closed** | 驗收通過，正式結案 |
| **Pending** | 暫停（有問題待解，或優先級調整） |
| **Cancelled** | 取消（有更好方案，或資源不足） |

### 狀態：常見的 Scrum 作法

| 狀態 | 說明 |
| --- | --- |
| **To Do** | 已建立，但還沒開始 |
| **In Progress** | 正在進行中，尚未完成 |
| **Done** | 已完成，等待驗收 |

### 兩套狀態的對照

這兩套其實是**同一條流程**，差別在後段的細緻度。Scrum 的三段把任務推到「開發完成、待驗收」就結束，因為驗收與結案被收斂進 **Sprint Review** 與 **DoD（Definition of Done）** 裡處理。

| 我的習慣 | Scrum 作法 | 說明 |
| --- | --- | --- |
| Open | To Do | 已建立、尚未開始，完全對應 |
| On Going | In Progress | 進行中，完全對應 |
| Done | Done | 開發完成、待驗收；但我的 Done 只是中段而非終點 |
| Verified | *（收斂進 DoD / Review）* | 驗收通過，Scrum 不另立狀態 |
| Closed | *（收斂進 DoD / Review）* | 正式結案，Scrum 不另立狀態 |

換句話說，我多出來的 **Verified** 與 **Closed**，是把 Scrum 壓在「Done」裡的「驗收」與「結案」獨立拉出來成為可追蹤的狀態。代價是流程看起來變長，好處是「完成 → 驗收」「驗收 → 上線」這兩段耗時能被量化，瓶頸才看得見。

> Pending 與 Cancelled 兩套作法都不算正常往前的流程節點，而是「暫時離開」或「退出」流動的旁支狀態。

### 為什麼要這麼多時間欄位

對使用者而言，最關心的一定是**上線時間**；但對 PM 來說，光是把完成時間分成「**預計**」與「**實際**」，就是為了在長期推進時，讓團隊慢慢縮小兩者的差距，使未來的規劃更準確。

把時間欄位攤開後，真正的價值在於**反推每一段的耗時，找出流程瓶頸**：

| 區間 | 反映的問題 |
| --- | --- |
| **建立 → 開始**（Open → On Going） | 任務排隊、資源不足或優先級不清；等待越久代表前置壅塞越嚴重 |
| **開始 → 完成**（實際完成起訖） | 執行的工作量與估時準確度；與「預計」對照即估時誤差來源 |
| **完成 → 驗收**（Done → Verified） | 驗收方的負荷；常被忽略卻最容易卡關（驗收人沒空、標準不清） |
| **驗收 → 上線**（Verified → 上線） | 上線審核、部署排程與發布窗口的限制 |

當你能看出「卡的不是開發、而是驗收」或「估時總是低估」這類**模式**時，這些時間欄位才從單純的紀錄，變成可以拿來**改善流程與校正估時**的依據。
