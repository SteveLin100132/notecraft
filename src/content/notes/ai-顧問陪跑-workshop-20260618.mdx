# **軟體開發流程與 QA 討論**

**日期／時間：** 2026 年 6 月 18 日上午 10:05（台北時間）

## **整體摘要**

團隊討論了軟體開發流程，重點包含任務分派、多 Session 處理、QA 與測試策略、Code Review，以及衝突解決。也討論到 API 版本管理、文件管理，以及建立明確規格與最佳實務的重要性，以維持開發與測試流程的一致性。

## **重點整理**

- 任務分派與多 Session 管理很重要，團隊討論如何透過 sub-agent 與主 Session 控制，改善平行處理並降低任務之間的干擾。
- QA 被視為需要獨立出來的必要流程，用來驗證任務是否完成，包含 Unit Test、End-to-End Test 與 Code Review；也建議建立檢查清單，甚至由專門的 QA sub-agent 負責。
- PR 與衝突處理流程中，團隊提到可先在本地端解決衝突，再推上 GitHub；同時也注意到多分支與合併管理上的挑戰。
- API 版本策略上，新 API 可採用 V2，同時保留 V1 以維持相容性，特別是 App 或舊版使用者仍可能依賴舊版本。
- 文件與規格管理很重要，團隊討論了 template、共用文件，以及統一 coding style、interface 與 best practice 的必要性。
- 專案文件中的圖片若要正常存取，可能需要公開連結，但這會帶來安全疑慮。
- 團隊正在使用 AI 助理自動化 issue 建立、標籤分類與工作流程管理，以提升效率。
- 團隊計畫使用 GitHub milestones 與 labels 來改善任務追蹤與優先順序管理。
- 需要釐清文件產出與 issue 拆分的負責人，避免多人同時處理造成混亂。
- 討論到系統擴展性與分散式架構，包括 Kubernetes、AWS、GCP 等雲端服務。

## **行動項目**

- 整合並優化任務分派與 Session 管理流程，支援平行開發並減少衝突。
- 建立獨立 QA 流程，明確定義檢查清單與負責範圍，並評估是否由專門 sub-agent 執行。
- 制定標準化 PR 與 Code Review 流程，包含本地端衝突解決與 AI 輔助建立 PR。
- 維護並更新 API 版本策略，確保新舊版本能平順過渡並保有向後相容性。
- 建立統一文件與 coding standards，提升團隊實作一致性。
- 指派特定成員負責文件產出與 issue 管理。
- 整合 AI 工具，自動化 issue 標籤與工作流程。
- 建立標籤與 milestone 的標準。
- 先規劃簡化版 MVP 工作流程，避免一開始過度複雜化。

- 如何設計多角色、多 Session 的任務分派，才能兼顧效率與穩定性？

- API 版本淘汰與遷移策略應如何規劃？
- 在 commit、review、衝突解決等流程中，AI 自動化與人工把關之間應如何取捨？
